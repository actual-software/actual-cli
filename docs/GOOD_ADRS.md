# ADR Quality Criteria and Authoring Guidelines

> Analysis of ~80 existing ADRs across 5 organizations, identifying what works, what doesn't, and what generic TypeScript/Next.js ADRs should look like.

## ADR Schema (from `packages/rules-lib/src/types.ts`)

Each ADR has these fields:

| Field | Type | Purpose |
|---|---|---|
| `id` | string (UUID) | Unique identifier |
| `title` | string | Short prescriptive name |
| `date` | string (ISO) | When the ADR was created |
| `context` | string | WHY this ADR exists (provenance, not tutorial) |
| `policies` | string[] | Canonical rules — the enforceable core |
| `instructions` | string[] | Optional how-to guidance (code patterns, migration steps) |
| `status` | enum | `created-from-user`, `created-from-suggestion`, `suggested`, etc. |
| `category` | object | `{ id, name, path }` from taxonomy |
| `scope` | object | Which repos/paths this ADR applies to |

## Two Generations Found in Production Data

### Generation 1: "Reverse-Engineered" ADRs (Bad — the majority)

These were AI-generated by analyzing existing codebases and describing what the code already does. They make up ~90% of the sample.

**Characteristics:**
- Titles are noun-phrase descriptions: "Adopt Modular Internal Library Architecture for Core Functionality"
- Context references specific files, line numbers, confidence percentages, and commit authors
- Policies describe existing patterns rather than prescribing rules
- Instructions are misused as "Positive/Negative" trade-off lists
- Extremely verbose — single policies are paragraph-length
- Massive duplication (6+ variations of "use OpenTelemetry for tracing" across one org)

**Example (bad):**

```
Title: "Adopt Structured Tracing for Performance Profiling and Observability"

Context: "Alacritty is a high-performance terminal emulator where rendering
latency and responsiveness are critical... The pattern emerged across search
functionality, display rendering, and event handling - all performance-critical
paths that require careful monitoring..."

Policy: "Implement structured tracing instrumentation across performance-critical
code paths using a tracing framework (likely the Rust `tracing` crate). This
involves instrumenting functions and code sections with trace spans that capture
timing, context, and hierarchical relationships between operations. The tracing
infrastructure is integrated into core components: search functionality
(alacritty_terminal/src/term/search.rs), display rendering
(alacritty/src/display/mod.rs), and event handling (alacritty/src/event.rs)..."

Instructions:
  - "Positive: Enables detailed performance profiling with minimal code changes..."
  - "Negative: Adds a dependency on a tracing framework..."
```

**Why this is bad:**
- Title doesn't tell you what to DO
- Context is a marketing paragraph about the project
- Policy describes what exists; doesn't prescribe a rule
- Instructions are trade-offs, not instructions
- References specific files in a specific project — useless as a generic ADR
- This exact ADR appears 6 times with slightly different wording

### Generation 2: Prescriptive ADRs (Good — the minority)

Found mostly from org `341e3c67` (Ethan Byrd's project). These are closer to what a generic ADR bank should contain.

**Characteristics:**
- Titles are prescriptive verb phrases: "Use TypeScript with Strict Type Checking"
- Context is 1-2 sentences citing concrete evidence
- Policies are single atomic rules
- Instructions are actual how-to steps
- Short and scannable

**Example (good):**

```
Title: "Hash Passwords with bcrypt Cost Factor 10"

Context: "Evidence from app/auth.ts:3 and app/app/api/register/route.ts:27
shows consistent use of bcrypt with cost factor 10 for password hashing."

Policy: "Hash all user passwords with bcrypt using cost factor 10 before
storing in database"

Instructions:
  - "Import hash function from 'bcrypt' for password registration"
  - "Use 'await hash(password, 10)' to hash passwords with cost factor 10"
  - "Import compare function from 'bcrypt' for password verification"
  - "Use 'await compare(password, user.hashedPassword)' to verify credentials"
  - "Never store plaintext passwords in the database"
```

**Example (good):**

```
Title: "Use Next.js 15 App Router for Application Structure"

Context: "Evidence from package.json:21 shows Next.js ^16.0.3 (Next.js 15
family) with App Router architecture."

Policy: "Use Next.js 15+ App Router architecture for all new routes and pages"

Instructions:
  - "Create routes using the app/ directory structure"
  - "Use page.tsx for route pages and layout.tsx for shared layouts"
  - "Leverage React Server Components by default (use 'use client' only when needed)"
  - "Place API routes in app/api/ directory using route.ts files"
```

**Example (user-created, extremely terse but valid):**

```
Title: "Align Components With Design System Repository"

Context: null

Policy: "Every component needs spec alignment with a component in my
design system repository"

Instructions: null
```

## Anti-Patterns Catalog

| Anti-Pattern | Description | Example |
|---|---|---|
| **Instructions as trade-offs** | Using the instructions field for "Positive/Negative" analysis instead of actionable how-to steps | "Positive: Reduces code duplication..." / "Negative: Learning curve..." |
| **Policy as description** | Describing what a codebase does rather than prescribing what to do | "The codebase adopts file-based storage as the primary datastore..." |
| **Title as noun phrase** | Naming the ADR like a research paper instead of a directive | "Adoption of Modular Internal Library Architecture for Core Functionality" |
| **Massive duplication** | Multiple ADRs saying the same thing with slightly different words | 6 variations of "use OpenTelemetry for tracing" in one org |
| **Codebase-specific context** | Context that only makes sense in one project | "Pattern detected across 23 files with 89.70% confidence" |
| **Wrong categories** | AI categorizing ADRs into unrelated taxonomy nodes | "Input Validation for AI Model Integration" under "Design System & Theming" |
| **No version info** | Failing to specify which framework/tool versions the ADR applies to | "Use Next.js App Router" (which version?) |
| **Mega-policy** | A single policy entry that is a full paragraph covering multiple decisions | Policies that contain "(1)... (2)... (3)... (4)... (5)..." numbered sub-decisions |
| **Framework marketing** | Context that reads like a framework's landing page | "React Native with Expo provides streamlined development experience with hot reloading, OTA updates, and simplified build process" |
| **Paraphrasing docs** | Restating official documentation without adding decision value | "NextResponse.json() returns JSON responses with appropriate HTTP status codes" |

## Quality Criteria for Good ADRs

### 1. Actionable

A developer reading the ADR knows exactly what to do or not do. The policy is a rule, not a description.

- **Pass**: "Use `'use client'` directive only in components that use hooks, event handlers, or browser APIs"
- **Fail**: "The application adopts a server-first rendering approach with selective client-side hydration"

### 2. Specific

References concrete APIs, patterns, file paths, or configuration keys. Not vague principles.

- **Pass**: "Configure `session: { strategy: 'jwt' }` in NextAuth configuration"
- **Fail**: "Use appropriate session management strategies"

### 3. Non-Obvious

Captures decisions that aren't immediately apparent from reading the framework's getting-started guide. Worth encoding because a developer might reasonably do it differently.

- **Pass**: "Co-locate server actions in the same file as the form component that calls them, not in a separate `actions/` directory"
- **Fail**: "Use TypeScript for type safety" (this is obvious for a TS project)

### 4. Versioned

Specifies which framework/tool versions the ADR applies to, since best practices change across major versions.

- **Pass**: "Use Next.js 15+ App Router architecture for all new routes and pages"
- **Fail**: "Use the App Router"

### 5. Testable

In principle, compliance can be verified — by a linter, a code reviewer, or an AI agent.

- **Pass**: "All API route handlers must return `NextResponse.json()` with explicit status codes"
- **Fail**: "Ensure API responses are well-structured"

### 6. Atomic

One decision per ADR. If a policy contains "(1)... (2)... (3)...", it should probably be split.

- **Pass**: One ADR for "Use Prisma for database access", another for "Use PostgreSQL as primary database"
- **Fail**: One mega-ADR covering ORM choice, database choice, migration strategy, and query patterns

## Structure Specification

### Title

Short, prescriptive. Verb phrase format: "Use X for Y" or "Prefer X over Y".

| Good | Bad |
|---|---|
| "Use Server Actions for Form Mutations" | "Adopt Server-Side Data Mutation Pattern Using Next.js Server Actions" |
| "Validate API Inputs with Zod Schemas" | "Implement Comprehensive Input Validation Strategy" |
| "Prefer `redirect()` over `router.push()` in Server Components" | "Adoption of Server-Side Navigation Patterns" |

### Context (1-3 sentences)

Explains WHY this ADR exists. Provenance, not tutorial. Should answer: "What problem does this solve?" or "What mistake does this prevent?"

| Good | Bad |
|---|---|
| "Next.js 15 defaults to `dynamic` rendering. Without explicit `export const dynamic = 'force-static'`, pages that could be statically generated will be rendered on every request, wasting compute." | "The application requires a robust rendering strategy that balances performance with dynamic content needs. As the codebase grew, the need for a consistent approach to rendering became critical..." |

### Policies (1-3 atomic rules)

The enforceable core. What an AI coding agent should check for and enforce. Each entry is one rule.

**Guidelines:**
- Use imperative mood: "Use...", "Prefer...", "Never...", "Always..."
- One rule per array entry
- Keep under 2 sentences per entry
- If you need more than 3 policies, split into multiple ADRs

### Instructions (optional, 2-6 concrete steps)

How-to guidance. Code patterns, commands, file paths. Things a developer would need to actually implement the policy.

**Guidelines:**
- Each entry is a concrete step or code pattern
- NOT trade-off analysis ("Positive: ...", "Negative: ...")
- Can include code snippets, terminal commands, file path conventions
- Skip if the policy is self-explanatory

## Granularity Guidelines

### Right-Sized ADR

- **One decision per ADR**: "Use Prisma for database access" is one ADR
- **3-5 policies max**: If you need more, split
- **Independently followable**: A developer can comply with this ADR without reading any other ADR

### Too Big (split it)

- 10+ policies covering an entire feature area
- Covers ORM choice AND migration strategy AND query patterns
- Title requires "and" — "Use Prisma for Database Access and Migration Management"

### Too Small (merge it or skip it)

- Single trivial policy that's obvious from the framework defaults
- "Use `.tsx` extension for React components" — this is a TypeScript default, not a decision

## Adapting for Generic ADRs

The existing ADRs in production are **specific** — they describe patterns found in real codebases. The ADRs we need for the bank are **generic** — they encode best practices for a technology stack that apply to any project using that stack.

Key differences:

| Aspect | Specific ADR | Generic ADR |
|---|---|---|
| **Context** | "Evidence from app/auth.ts:3 shows bcrypt usage" | "Password storage requires one-way hashing to protect credentials if the database is compromised" |
| **Policy** | "Hash passwords with bcrypt cost factor 10 as implemented in auth.ts" | "Hash all user passwords with bcrypt (cost factor ≥10) before database storage" |
| **Instructions** | References project-specific file paths | References framework conventions and common patterns |
| **Scope** | Tied to specific repos and file patterns | Applies to any project using the relevant stack |
| **Versioning** | May reference exact package.json versions | Specifies major version ranges (e.g., "Next.js 14+") |

### What to Keep from Specific ADRs

- The decision itself (e.g., "use JWT for sessions")
- Concrete how-to instructions (e.g., "configure `session: { strategy: 'jwt' }`")
- Version specificity (e.g., "NextAuth v5")

### What to Drop

- File path references to specific projects
- Confidence scores and pattern detection metadata
- Author attribution and commit hashes
- "Positive/Negative" trade-off lists in instructions
- Paragraph-length policies

## Appendix: Best Examples from Production Data

These are the ADRs from production that most closely match the quality criteria above. They serve as templates for generic ADR authoring.

### Template A: Framework Configuration Decision

```json
{
  "title": "Use Next.js 15 App Router for Application Structure",
  "context": "Next.js 15 introduces a stable App Router with React Server Components, streaming, and file-based routing. New projects should use App Router over the legacy Pages Router.",
  "policies": [
    "Use Next.js 15+ App Router architecture for all new routes and pages"
  ],
  "instructions": [
    "Create routes using the app/ directory structure",
    "Use page.tsx for route pages and layout.tsx for shared layouts",
    "Leverage React Server Components by default (use 'use client' only when needed)",
    "Place API routes in app/api/ directory using route.ts files"
  ]
}
```

### Template B: Security Decision

```json
{
  "title": "Protect Cron Endpoints with CRON_SECRET Environment Variable",
  "context": "Cron job API endpoints are publicly accessible URLs. Without authentication, anyone who discovers the endpoint can trigger scheduled jobs.",
  "policies": [
    "Protect all cron job API endpoints with CRON_SECRET authorization check using Bearer token validation"
  ],
  "instructions": [
    "Set CRON_SECRET environment variable with a secure random string",
    "Extract authorization header from request: request.headers.get('authorization')",
    "Validate header matches: authHeader === `Bearer ${process.env.CRON_SECRET}`",
    "Return 401 Unauthorized if validation fails",
    "Configure external cron services (e.g., Vercel Cron) to include Authorization header"
  ]
}
```

### Template C: ORM/Data Access Decision

```json
{
  "title": "Use Prisma Client for Type-Safe Database Access",
  "context": "Direct SQL queries lose type safety and require manual type definitions. Prisma generates TypeScript types from the schema, catching query errors at compile time.",
  "policies": [
    "Use Prisma Client for all database operations to ensure type-safe database access across the application"
  ],
  "instructions": [
    "Import Prisma Client from '@prisma/client'",
    "Use the singleton prisma instance from '@/lib/prisma' in all database operations",
    "Leverage Prisma's type generation for compile-time safety",
    "Use Prisma's query API (findMany, create, update, delete) instead of raw SQL"
  ]
}
```

### Template D: Authentication Decision

```json
{
  "title": "Use JWT Strategy for Session Management",
  "context": "JWT-based sessions are stateless, eliminating server-side session storage and enabling horizontal scaling without session affinity.",
  "policies": [
    "Use JWT strategy for session management to enable stateless authentication"
  ],
  "instructions": [
    "Configure 'session: { strategy: \"jwt\" }' in NextAuth configuration",
    "Implement session callbacks to enrich JWT tokens with user data",
    "Use getServerSession() or auth() to access session in server components",
    "Configure appropriate JWT expiration times"
  ]
}
```
