# ADR Quality Criteria and Authoring Guidelines

> General-purpose guide for writing effective Architecture Decision Records for use
> with AI coding agents (Claude Code, Cursor, Codex) via AGENTS.md / CLAUDE.md files.
> Language- and framework-agnostic. The schema, taxonomy, and category paths below
> are fixed — do not add or rename fields.

---

## ADR Schema (from `packages/rules-lib/src/types.ts`)

Each ADR must be a JSON object with **exactly** these nine fields. No more, no less.
Downstream tooling reads this schema and will reject or misparse records with missing,
renamed, or extra fields.

| Field | Type | Purpose |
|---|---|---|
| `id` | `string` (UUID) | Unique identifier |
| `title` | `string` | Short prescriptive name |
| `date` | `string` (ISO) | When the ADR was created |
| `context` | `string` | WHY this ADR exists (provenance, not tutorial) |
| `policies` | `string[]` | Canonical rules — the enforceable core |
| `instructions` | `string[]` | Optional how-to guidance (code patterns, migration steps) |
| `status` | `enum` | `created-from-user` \| `created-from-suggestion` \| `suggested` |
| `category` | `object` | `{ id, name, path }` from taxonomy |
| `scope` | `object` | Which repos/paths this ADR applies to |

**NEVER add extra fields** (e.g. `applies_to_languages`, `version`, `rationale`, `decision`).
**NEVER omit any of the nine fields** — all are required.
**NEVER rename fields** — use `policies` not `rules`, `instructions` not `steps`, `context` not `background`.

## Two Generations Found in Production Data

### Generation 1: "Reverse-Engineered" ADRs (Bad — the majority)

AI-generated by analyzing existing codebases and describing what the code already does.
These make up ~90% of observed production samples.

**Characteristics:**
- Titles are noun-phrase descriptions, not directives
- Context references specific files, line numbers, confidence percentages, and commit authors
- Policies describe existing patterns rather than prescribing rules
- Instructions misused as "Positive / Negative" trade-off lists
- Extremely verbose — single policies run to full paragraphs
- Massive duplication (6+ variations of the same ADR across one org)

**Example (bad):**

```
Title: "Adopt Structured Tracing for Performance Profiling and Observability"

Context: "The pattern emerged across search functionality, display rendering, and
event handling — all performance-critical paths that require careful monitoring..."

Policy: "Implement structured tracing instrumentation across performance-critical
code paths using a tracing framework. This involves instrumenting functions and
code sections with trace spans that capture timing, context, and hierarchical
relationships between operations..."

Instructions:
  - "Positive: Enables detailed performance profiling with minimal code changes..."
  - "Negative: Adds a dependency on a tracing framework..."
```

**Why this is bad:**
- Title doesn't tell you what to do
- Policy describes what exists; doesn't prescribe a rule
- Instructions are trade-offs, not steps
- Could apply to any project in any language — adds no decision value
- This exact ADR appears 6 times with slightly different wording

---

### Generation 2: Prescriptive ADRs (Good — the minority)

**Characteristics:**
- Titles are prescriptive verb phrases: "Use X for Y" or "Prefer X over Y"
- Context is 1–2 sentences explaining the WHY
- Policies are single, atomic, imperative rules
- Instructions are concrete how-to steps
- Short and scannable

**Example (good):**

```json
{
  "id": "a1b2c3d4-...",
  "title": "Hash Passwords with bcrypt Cost Factor ≥ 10",
  "date": "2025-01-15",
  "context": "Password storage requires one-way hashing. If the database is compromised, plaintext passwords expose all accounts immediately. bcrypt's cost factor controls iteration count and therefore brute-force resistance.",
  "policies": [
    "Hash all user passwords with bcrypt at cost factor ≥ 10 before storing in the database",
    "Never store plaintext or reversibly-encrypted passwords"
  ],
  "instructions": [
    "Import hash and compare from your bcrypt library",
    "Use `await hash(password, 10)` on registration",
    "Use `await compare(plaintext, storedHash)` on login — never compare plaintext directly"
  ],
  "status": "created-from-suggestion",
  "category": {
    "id": "security",
    "name": "Security",
    "path": "Backend > Security"
  },
  "scope": {
    "repositories": [],
    "paths": []
  }
}
```

## Anti-Patterns Catalog

| Anti-Pattern | Description | Instead |
|---|---|---|
| **Instructions as trade-offs** | Using the instructions field for Positive/Negative analysis | Use instructions for concrete steps: "Run `npm install bcrypt`", not "Positive: adds security" |
| **Policy as description** | Describing what a codebase does rather than prescribing what to do | Write "Use X for Y" not "The codebase uses X for Y" |
| **Title as noun phrase** | Naming the ADR like a research paper instead of a directive | Write "Validate Request Inputs with Zod" not "Adoption of Zod for Input Validation" |
| **Massive duplication** | Multiple ADRs saying the same thing with slightly different words | Merge into one canonical ADR — check the existing list before adding |
| **Codebase-specific context** | Context that only makes sense in one specific project | Write generic context: "Without X, Y fails" not "Found in 23 files with 89% confidence" |
| **Wrong category** | ADRs placed in unrelated taxonomy nodes | Verify the category path makes sense; do not invent new categories |
| **No version info** | Failing to specify which framework/tool versions apply | Always state the major version: "Next.js 13+" not "the new App Router" |
| **Mega-policy** | A single policy entry covering multiple decisions in a paragraph | One rule per entry — if you write "(1)... (2)...", split into separate policies |
| **Framework marketing** | Context that reads like a landing page, not a decision record | Write what goes wrong without the decision, not why the technology is great |
| **Paraphrasing docs** | Restating official documentation without adding decision value | State the non-obvious: what the default is, why it's wrong, what to do instead |
| **Obvious from getting started** | Decision is the framework's default, taught in every intro tutorial | Skip it — only include decisions a competent developer might reasonably get wrong |
| **Too granular** | Micro-decision affecting <5% of projects or pure preference | Skip it — only include decisions that apply to most projects using this stack |
| **Negation without alternative** | Saying what NOT to do without specifying the correct replacement | "Do not use raw SQL. Use the ORM client from `@/lib/db` for all queries." |
| **Version ambiguity** | Citing a framework feature without stating the version it applies to | "Use the new cache API (framework v14+)" |
| **Config without context** | Showing a configuration snippet without explaining why it matters | Explain what goes wrong if the default is kept: "`strict: true` alone does not catch index-access errors — `noUncheckedIndexedAccess` must be added explicitly" |

## Quality Criteria for Good ADRs

### 1. Actionable

A developer reading the ADR knows exactly what to do or not do. The policy is a rule, not a description.

- **Pass**: "Use the `'use client'` directive only in components that use hooks, event handlers, or browser APIs"
- **Fail**: "The application adopts a server-first rendering approach with selective client-side hydration"

### 2. Specific

References concrete APIs, config keys, file paths, or CLI flags. Not vague principles.

- **Pass**: "Configure `session: { strategy: 'jwt' }` in the auth configuration"
- **Fail**: "Use appropriate session management strategies"

### 3. Non-Obvious

Captures decisions that aren't immediately apparent from the framework's getting-started guide. Worth encoding because a developer might reasonably do it differently.

- **Pass**: "Wrap data-access functions with `cache()` when using an ORM — ORMs are not auto-memoized the way `fetch` is"
- **Fail**: "Use TypeScript for type safety" (obvious for a TypeScript project)

### 4. Versioned

Specifies which framework/tool versions the ADR applies to, since best practices change across major versions.

- **Pass**: "Use the App Router architecture for all new routes (Next.js 13+)"
- **Fail**: "Use the App Router"

### 5. Testable

In principle, compliance can be verified — by a linter, a code reviewer, or an AI agent.

- **Pass**: "All API route handlers must return explicit HTTP status codes"
- **Fail**: "Ensure API responses are well-structured"

### 6. Atomic

One decision per ADR. If a policy contains "(1)... (2)... (3)...", it should be split.

- **Pass**: One ADR for "Use Prisma for database access", a separate ADR for "Use PostgreSQL as the primary database"
- **Fail**: One mega-ADR covering ORM choice, database choice, migration strategy, and query patterns

## Structure Specification

### Word Count

**Aim for 200 words. Hard cap: 220 words** (title excluded; count context + all policies + all instructions combined).

If an ADR exceeds 220 words, cut it — do not pad with qualifiers or restate the policy in context. If it cannot fit in 220 words without losing its meaning, it covers too many decisions and should be split.

### Title

Short, prescriptive. Verb phrase format.

| Good | Bad |
|---|---|
| "Validate Request Inputs with Zod Schemas" | "Implement Comprehensive Input Validation Strategy" |
| "Prefer `redirect()` over `router.push()` in Server Components" | "Adoption of Server-Side Navigation Patterns" |
| "Use Dependency Injection for External Service Clients" | "Modular Service Architecture with Dependency Management" |

**NEVER** write a title as a noun phrase. If the title does not start with a verb, rewrite it.

### Context (1–3 sentences)

Explains WHY this ADR exists. Answers: "What problem does this solve?" or "What mistake does this prevent?" Value-neutral — describes forces, not marketing copy.

| Good | Bad |
|---|---|
| "Without explicit opt-out, statically-renderable pages are re-rendered on every request, wasting compute." | "The application requires a robust rendering strategy that balances performance with dynamic content needs." |
| "ORMs are not auto-memoized. Without `cache()`, calling `getUser()` in both a layout and a page results in two database queries per request." | "We use an ORM for type-safe database access across all components." |

**NEVER** write context that references files, line numbers, commit authors, or confidence percentages.
**NEVER** write context that reads like a product pitch for the technology.

### Policies (1–3 atomic rules)

The enforceable core. Use imperative mood. One rule per array entry.

**Rules:**
- Start with "Use...", "Prefer...", "Never...", "Always...", or "Do not..."
- **NEVER** include sub-clauses or numbered sub-decisions in a single entry — one rule per entry
- Keep under 2 sentences per entry
- More than 3 policies → split into multiple ADRs. **NEVER** write 4+ policies in one ADR.

### Instructions (optional, 2–6 concrete steps)

How-to guidance: code patterns, CLI commands, file conventions.

**Rules:**
- Each entry is a concrete step or code snippet
- **NEVER** use instructions for trade-off analysis ("Positive: ...", "Negative: ..."). That belongs in `context`.
- Skip entirely if the policy is self-explanatory

## Granularity Guidelines

### Right-Sized ADR
- One decision per ADR
- 1–3 policies — if you need more, split
- Independently followable without reading other ADRs

### Too Big (split it)
- Covers multiple decisions (ORM choice AND migration strategy AND query patterns)
- Title requires "and"
- More than 3 policies

### Too Small (merge or skip)
- Single trivial policy obvious from framework defaults
- Pure-preference decision with no material impact on correctness or maintainability

## Generic vs Specific ADRs

ADRs in the bank are **generic** — they apply to any project using a given stack.

| Aspect | Specific ADR (bad) | Generic ADR (good) |
|---|---|---|
| **Context** | "Evidence from app/auth.ts:3 shows bcrypt usage" | "Password storage requires one-way hashing to protect credentials if the database is compromised" |
| **Policy** | "Hash passwords with bcrypt cost factor 10 as implemented in auth.ts" | "Hash all user passwords with bcrypt (cost factor ≥10) before database storage" |
| **Instructions** | References project-specific file paths | References framework conventions and common patterns |
| **Versioning** | Exact package.json versions | Major version ranges (e.g., "framework v14+") |

## Writing Effective AGENTS.md / CLAUDE.md Instructions Using ADRs

ADRs loaded into AGENTS.md and CLAUDE.md act as persistent rules for AI coding agents.
The way you express rules directly determines how reliably they are followed.

### Rule Phrasing Principles

**1. Explicit denial beats general suggestion.**
Vague discouragement is unreliable. Name the exact thing to avoid, then state the correct alternative.

| Less reliable | More reliable |
|---|---|
| "Avoid raw database queries" | "Do NOT use raw SQL. Use the ORM client from `@/lib/db` for all queries." |
| "Consider using structured error handling" | "Do NOT throw inside Server Actions for expected errors. Return `{ success: false, error: string }` instead." |
| "Try to use meaningful variable names" | "Name booleans with is/has/should prefix: `isLoading`, not `loading`." |

**2. Pair every prohibition with its correct alternative.**
A bare `NEVER do X` leaves the agent guessing what to do instead. Always specify the replacement.

- **Incomplete**: "NEVER push directly to main."
- **Complete**: "NEVER push directly to main — open a PR and wait for CI. Direct pushes bypass the review and test pipeline."

**3. Provide the WHY for non-negotiable rules.**
Context improves adherence because the agent generalizes the rule to similar situations.
Reserve WHY explanations for hard rules — do not pad every guideline with rationale.

**4. Use `NEVER` / `ALWAYS` / `IMPORTANT` sparingly, for hard rules only.**
If every line uses emphasis, none of them read as emphatic. Reserve these words for rules that must not be violated under any circumstances.

**5. Use positive framing for style and format rules.**
For non-safety rules (formatting, naming, structure), telling the agent what TO do is more reliable than what NOT to do.

| Less reliable | More reliable |
|---|---|
| "Do not write mega-policies." | "Each policy must be one atomic rule — one sentence, one array entry. Split if you need more than three." |
| "Don't use vague context." | "Context must state what goes wrong without this decision. One to three sentences." |

**6. Include a self-verification step.**
Agents perform measurably better when they can check their own output. Add a checklist.

```markdown
After generating candidate ADRs, verify each one:
- Does the title start with a verb?
- Does context explain WHY (not what)?
- Is each policy one atomic rule?
- Are instructions concrete steps (not trade-offs)?
- Is a framework version cited?
Reject any candidate that fails a check — do not include it in the output.
```

**7. Use XML tags for complex rule sets.**
Wrapping related rules in `<rules>` or `<schema>` tags helps the agent parse them unambiguously, especially in long files.

### Template: ADR rules block for AGENTS.md / CLAUDE.md

```markdown
## ADR Authoring Rules

When generating or editing ADRs, follow these rules exactly.

<schema>
Every ADR must include exactly these fields and NO OTHERS:
  id, title, date, context, policies, instructions, status, category, scope

NEVER add: applies_to_languages, version, rationale, decision, or any other field.
NEVER omit any of the nine fields.
NEVER rename fields (use "policies" not "rules", "instructions" not "steps").
</schema>

<title>
Write as a prescriptive verb phrase: "Use X for Y", "Prefer X over Y", "Never X without Y".
Do NOT write as a noun phrase: "Adoption of X", "X Management Strategy".
If the title does not start with a verb, rewrite it.
</title>

<context>
Write 1–3 sentences explaining WHY this decision matters — what goes wrong without it.
Do NOT describe what the code does.
Do NOT write marketing copy ("X provides streamlined development...").
Do NOT reference file paths, line numbers, or confidence percentages.
</context>

<policies>
Write 1–3 atomic imperative rules. Each entry is one rule in one sentence.
Do NOT write sub-clauses or numbered sub-decisions inside a single policy entry.
Do NOT write more than 3 policies — split into multiple ADRs instead.
Start each rule with: "Use...", "Prefer...", "Never...", "Always...", or "Do not...".
</policies>

<instructions>
Write 2–6 concrete how-to steps or code snippets. Each entry is one actionable step.
Do NOT write trade-off analysis ("Positive: ...", "Negative: ...") — that belongs in context.
Omit this field entirely if the policies are self-explanatory.
</instructions>

IMPORTANT: After generating candidates, verify each one is actionable, specific,
non-obvious, versioned (states framework version), testable, and atomic.
Reject and discard any candidate that fails a check — do not include it in the output.
```
