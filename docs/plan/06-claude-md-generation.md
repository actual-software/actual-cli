# 06 - CLAUDE.md Generation & Merge Strategy

## Overview

The final step of `actual adr-bot` writes the AI-generated CLAUDE.md content to disk. The CLI handles managed section markers, merging with existing files, and writing multiple files (root + subdirectory).

The markdown content itself is generated by Claude Code during the combined tailoring+formatting step (see [05-adr-tailoring.md](./05-adr-tailoring.md)). This module is responsible for:

1. Wrapping AI-generated content in managed section markers
2. Merging with existing CLAUDE.md files (preserving user content)
3. Writing files to disk
4. Handling the per-file user confirmation flow

## File Layout

Claude Code decides which files to create based on the project structure:

```
CLAUDE.md                    # Root-level: general/cross-cutting rules
apps/web/CLAUDE.md           # Next.js + React specific rules
apps/api/CLAUDE.md           # Fastify specific rules
```

- **Root CLAUDE.md**: Always generated. Contains general rules that apply across all projects (TypeScript conventions, commit style, etc.)
- **Subdirectory CLAUDE.md files**: Generated only when there are project-specific ADRs for that project. Not generated for projects that only have general/language-level rules.

## CLAUDE.md Format

The output format is **directive-style** -- terse, actionable instructions optimized for Claude's consumption. The markdown structure within the managed section is AI-generated (not a fixed template). Claude Code organizes content naturally using headings, bullet points, and inline code references.

### Example Output (Root)

```markdown
# Project Guidelines

<!-- managed:actual-start -->
<!-- last-synced: 2025-01-15T10:30:00Z -->
<!-- version: 1 -->

- Use TypeScript strict mode across all packages
- Use pnpm as the package manager; do not use npm or yarn
- Use conventional commits (feat:, fix:, chore:, etc.)
- Write unit tests for all business logic; colocate test files with source

<!-- managed:actual-end -->
```

### Example Output (Subdirectory)

```markdown
<!-- managed:actual-start -->
<!-- last-synced: 2025-01-15T10:30:00Z -->
<!-- version: 1 -->

## Architecture
- Use the App Router (`app/` directory) for all routing; do not create files in `pages/`
- Use server components by default; add `'use client'` only for components needing browser APIs
- Colocate page-specific components in the route directory (e.g., `app/dashboard/_components/`)

## Styling
- Use Tailwind CSS for all styling; do not use CSS modules or styled-components

## Testing
- Use Vitest + React Testing Library for component tests (`*.test.tsx`, colocated with source)

<!-- managed:actual-end -->
```

### Key Rules

1. **Managed section markers**: Content between `<!-- managed:actual-start -->` and `<!-- managed:actual-end -->` is owned by `actual`. Manual edits within this section will be overwritten on re-sync. Markers are HTML comments (invisible in rendered markdown).

2. **All files use markers**: Both root and subdirectory CLAUDE.md files use managed markers. User content outside markers is preserved in all files.

3. **Default header on new files**: When creating a new root CLAUDE.md, a default `# Project Guidelines` header is added above the managed section. Subdirectory CLAUDE.md files do not get a default header. Existing headers in any file are preserved.

4. **Metadata comments**: `last-synced` timestamp and `version` number as HTML comments within the managed section.

## Create vs. Update Logic

Applied independently to each file (root and subdirectory):

```
if file does not exist:
    -> Create new file with managed section (root gets default header)
else if file exists but has no managed markers:
    -> Append managed section to end of file
else if file exists with managed markers:
    -> Replace content between markers with new managed section
    -> Preserve everything outside markers
```

### Merge Algorithm

```rust
fn merge_claude_md(existing: &str, new_managed_content: &str, is_root: bool) -> String {
    let start_marker = "<!-- managed:actual-start -->";
    let end_marker = "<!-- managed:actual-end -->";

    if let (Some(start), Some(end)) = (
        existing.find(start_marker),
        existing.find(end_marker),
    ) {
        // Replace managed section
        let before = &existing[..start];
        let after = &existing[end + end_marker.len()..];
        format!("{before}{new_managed_content}{after}")
    } else {
        // Append managed section
        format!("{existing}\n\n{new_managed_content}\n")
    }
}

fn wrap_in_markers(content: &str, version: u32) -> String {
    let now = chrono::Utc::now().to_rfc3339();
    format!(
        "<!-- managed:actual-start -->\n\
         <!-- last-synced: {now} -->\n\
         <!-- version: {version} -->\n\n\
         {content}\n\n\
         <!-- managed:actual-end -->"
    )
}
```

### No Backup Files

No `.bak` files are created. Git is the backup mechanism. If the directory is not a git repo, the CLI warns but proceeds.

## User Confirmation (Per-File)

After tailoring+formatting, the CLI presents each generated file for review:

```
Generated CLAUDE.md files:

  [1] CLAUDE.md (root) -- 6 rules, general conventions
  [2] apps/web/CLAUDE.md -- 14 rules, Next.js + React
  [3] apps/api/CLAUDE.md -- 8 rules, Fastify

  Skipped 4 ADRs (not applicable).

Actions:
  [a]ccept all    [r]eject #    [p]review #    [q]uit
>
```

- **Accept all**: Write all files
- **Reject #**: Don't write that specific file (file-level rejection is NOT remembered across syncs; ADR-level rejection via `--reset-rejections` is separate)
- **Preview #**: Show the full rendered markdown content for a file
- **Quit**: Cancel entirely, write nothing

## Dry Run

`--dry-run` prints a summary without writing:

```
Dry run -- no files will be written.

  CLAUDE.md (root): would create (6 rules)
  apps/web/CLAUDE.md: would create (14 rules)
  apps/api/CLAUDE.md: would update (8 rules, replacing existing managed section)

  Skipped 4 ADRs.
```

Combine with `--full` to output the complete rendered content for all files to stdout.

## Re-sync Summary

When running on a repo with existing managed sections, the CLI shows what changed:

```
CLAUDE.md changes:

  CLAUDE.md (root):
    + 2 new rules, ~ 1 updated, - 0 removed

  apps/web/CLAUDE.md:
    + 3 new rules, ~ 0 updated, - 1 removed

  apps/api/CLAUDE.md: (new file)
    + 8 rules

[a]ccept all    [r]eject #    [p]review #    [q]uit
```

Change detection is approximate -- the CLI compares the new managed section content against the existing managed section content as text. Individual rule tracking relies on the `adr_ids` from the tailoring output.

## Version Tracking

The managed section includes a version comment:

```markdown
<!-- version: 1 -->
```

This increments on each sync per file. Combined with `last-synced` and git history, users can track when and how each CLAUDE.md was last modified.

## Rendering Pipeline

Since the markdown content is AI-generated, the "rendering" pipeline is simpler than a deterministic renderer:

```
tailoring_output: TailoringOutput (from Claude Code)
  |
  |-- For each file in tailoring_output.files:
  |   |-- Check user confirmation (accept/reject)
  |   |-- If accepted:
  |   |   |-- Wrap content in managed markers (add metadata comments)
  |   |   |-- Read existing file (if any)
  |   |   |-- Merge (replace managed section or append)
  |   |   |-- Write to disk
  |   |-- If rejected:
  |       |-- Skip
  |
  |-- Report telemetry (adrs_written count)
```

## Future: Multi-Provider Support

v1 targets CLAUDE.md only. Planned future providers and their target files:

| Provider | File | Status |
|----------|------|--------|
| Claude Code | `CLAUDE.md` | v1 |
| Cursor | `.cursorrules` or `RULES.md` | Planned |
| Codex (OpenAI) | `AGENT.md` | Planned |
| Copilot | `.github/copilot-instructions.md` | Planned |

The architecture supports adding providers by:
1. Changing the tailoring prompt to request provider-specific output formats
2. Adjusting the output schema's `path` field to support different file locations
3. The merge logic and markers work the same regardless of the target file
