### Rules

**Note:** The source ADR does not contain explicit R-PREFIX-### rule identifiers. The following rules are derived from the ADR's policy statements:

- **R-PUB-001** MUST: Implement protocol-based public API design pattern that defines explicit contracts for external interactions.
- **R-PUB-002** MUST: Establish formal protocols/interfaces that serve as the public API surface, separating contract from implementation details.
- **R-PUB-003** MUST: Apply the pattern consistently across integration test scenarios (no tailor, sync modes, smoke tests).
- **R-PUB-004** MUST: Ensure external consumers interact through well-defined protocol boundaries rather than directly with internal implementations.
- **R-PUB-005** SHOULD: Maintain clear separation between public API contracts and internal implementation to enable independent evolution.
- **R-PUB-006** SHOULD: Use protocol-based integration tests that validate external-facing behavior.
- **R-PUB-007** SHOULD: Design protocol boundaries to provide backward compatibility guarantees for external API consumers.
- **R-PUB-008** SHOULD: Maintain discipline to avoid leaking implementation details through protocol boundaries.

### Context

**Positive aspects:**
- Clear separation between public API contracts and internal implementation
- Enhanced testability through protocol-based integration tests
- Improved backward compatibility guarantees
- Better documentation surface as protocols explicitly define the public contract
- Reduced coupling between external consumers and internal implementation

**Trade-offs:**
- Additional abstraction layer introduces slight complexity overhead
- Requires discipline to maintain protocol boundaries
- May require more upfront design effort

**Limitations:**
- Changes to public protocols require careful versioning and migration strategies
