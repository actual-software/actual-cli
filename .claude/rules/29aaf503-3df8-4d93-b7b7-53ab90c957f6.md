## Context

The codebase requires consistent testing and benchmarking capabilities across multiple components. Three files (benches/analysis_baseline.rs, tests/integration_errors.rs, and src/testutil.rs) demonstrate a pattern of utilizing core library modules to support testing and performance analysis infrastructure. This pattern emerged from the need to maintain separation of concerns between production code, test utilities, and performance benchmarks while ensuring code reusability and maintainability.

### Rules

- **R-MTBI-001** MUST: Organize core functionality into reusable library modules that can be consumed by benchmarks, tests, and test utilities.
- **R-MTBI-002** MUST: Maintain clear separation between production code (src/), test utilities (src/testutil.rs), integration tests (tests/), and performance benchmarks (benches/).
- **R-MTBI-003** MUST: Centralize common test utilities in dedicated modules (such as src/testutil.rs) that can be shared across different testing contexts.
- **R-MTBI-004** MUST: Avoid circular dependencies between test utilities and production code.
- **R-MTBI-005** SHOULD: Feature-gate test utilities in src/ directory to avoid increasing compilation time for production builds.
- **R-MTBI-006** SHOULD: Use standardized utility functions from centralized test modules to ensure consistent testing patterns across the codebase.
- **R-MTBI-007** MAY: Import and utilize core library modules as needed from benchmarks, integration tests, and test utilities.

**Accept when:**
- Production code, test utilities, integration tests, and benchmarks are clearly separated into src/, src/testutil.rs, tests/, and benches/ directories respectively
- Common test utilities are centralized in dedicated modules accessible to both unit and integration tests
- No circular dependencies exist between test utilities and production code
- Test utilities in src/ are appropriately feature-gated (if applicable)
- Core library modules can be imported and utilized by benchmarks, tests, and test utilities
- Testing patterns are consistent across the codebase through use of standardized utility functions

Claude Code MUST verify adherence to this modular architecture when creating or modifying test utilities, benchmarks, or integration tests. Verification MUST NOT be skipped or deferred.
</enforcement>

## Consequences

### Positive
- Clear separation of concerns between production code, test infrastructure, and benchmarking code
- Improved code reusability through shared test utility modules accessible to both unit and integration tests
- Easier maintenance as test-related functionality is centralized in dedicated modules
- Better discoverability of testing utilities for developers working on the codebase
- Consistent testing patterns across the codebase through standardized utility functions

### Negative
- Additional complexity in module organization and dependency management
- Potential for circular dependencies if not carefully managed between test utilities and production code
- Test utilities in src/ directory may increase compilation time for production builds if not properly feature-gated
