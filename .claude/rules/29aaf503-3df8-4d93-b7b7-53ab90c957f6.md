These rules are ALWAYS ACTIVE for all files in the codebase to ensure consistent modular architecture for test and benchmark infrastructure.
</rule_activation>

### Rules

- **R-MLAT-001** MUST: Implement a modular library architecture where core functionality is organized into reusable library modules that can be consumed by benchmarks, tests, and test utilities.
- **R-MLAT-002** MUST: Maintain clear separation of concerns between production code (src/), test utilities (src/testutil.rs), integration tests (tests/), and performance benchmarks (benches/).
- **R-MLAT-003** MUST: Centralize common test utilities in dedicated modules (e.g., src/testutil.rs) that can be shared across different testing contexts.
- **R-MLAT-004** MUST: Each component (benchmarks, tests, test utilities) should import and utilize core library modules as needed.
- **R-MLAT-005** SHOULD: Avoid circular dependencies between test utilities and production code through careful module organization.
- **R-MLAT-006** SHOULD: Feature-gate test utilities in src/ directory to prevent increased compilation time for production builds.
- **R-MLAT-007** SHOULD: Follow consistent testing patterns across the codebase through standardized utility functions.
- **R-MLAT-008** MAY: Introduce additional test utility modules as needed to support specific testing scenarios while maintaining architectural consistency.

### Verify

```bash
# Note: The source ADR does not contain an explicit verify section.
# The following verification commands check the architectural structure:

# Verify test utilities are in src/testutil.rs or similar dedicated modules
find src -name "*testutil*" -o -name "*test_util*" 2>/dev/null | head -20

# Check for clear separation of concerns
ls -la src/ tests/ benches/ 2>/dev/null | head -30

# Verify benchmarks are in benches/
find benches -name "*.rs" 2>/dev/null | head -10

# Verify integration tests are in tests/
find tests -name "*.rs" 2>/dev/null | head -10

# Check for potential circular dependencies (basic check)
# This looks for test utilities importing from tests or benches
if [ -f src/testutil.rs ]; then
  grep -n "use.*tests::" src/testutil.rs 2>/dev/null && echo "Warning: Potential circular dependency detected" || echo "No obvious circular dependencies in testutil"
fi
```

**Accept when:**
- Clear directory separation exists between src/, tests/, and benches/
- Test utilities are centralized in dedicated modules (e.g., src/testutil.rs)
- Benchmarks (benches/*.rs) import from core library modules
- Integration tests (tests/*.rs) import from core library modules and test utilities
- Test utilities are shared across unit tests and integration tests
- No circular dependencies exist between test utilities and production code
- Test utility modules are properly feature-gated or structured to minimize production build impact
