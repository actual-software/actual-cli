### Rules

**Note:** The source ADR does not contain explicit R-PREFIX-### rule IDs. The following rules are derived from the ADR's policies:

- **R-SEC-001** MUST NOT: Hardcode credentials, API keys, or secrets in configuration files or source code
- **R-SEC-002** MUST: Use environment variables, encrypted secret stores, or dedicated secrets management services (HashiCorp Vault, AWS Secrets Manager, or CI platform-native solutions) for credential storage
- **R-SEC-003** MUST: Inject credentials at runtime rather than embedding them in code
- **R-SEC-004** MUST: Apply consistent secrets handling pattern across CLI commands and testing infrastructure
- **R-SEC-005** MUST: Ensure config.rs, status.rs, and e2e_live.rs retrieve secrets from secure stores at runtime
- **R-SEC-006** SHOULD: Separate development and production secret access to prevent developers from requiring production credentials

### Verify

**Note:** The source ADR does not contain explicit verification commands. The following verification approach is recommended:

```bash
# Check for hardcoded secrets patterns in source code
grep -r -i -E '(password|api[_-]?key|secret|token|credential)\s*=\s*["\047][^"\047]+["\047]' --include="*.rs" --include="*.toml" --include="*.json" --include="*.yaml" --include="*.yml" . || echo "No hardcoded secrets detected"

# Verify config.rs, status.rs, and e2e_live.rs use environment variables or runtime injection
for file in config.rs status.rs e2e_live.rs; do
  if [ -f "$file" ]; then
    echo "Checking $file for runtime secret retrieval..."
    grep -E '(env::var|std::env::var|get_env)' "$file" && echo "✓ $file uses environment variables" || echo "✗ $file may need review"
  fi
done

# Check for secrets in version control history (basic check)
git log --all --full-history --source --find-object=$(git hash-object /dev/null) --grep='password\|secret\|key' --oneline | head -20 || echo "No obvious secrets in recent git history"
```

**Accept when:**
- No credentials are hardcoded in configuration files (*.toml, *.json, *.yaml, *.yml) or source code (*.rs)
- config.rs, status.rs, and e2e_live.rs retrieve secrets from environment variables or secrets management services at runtime
- Consistent pattern of runtime secret injection is applied across all CLI commands and testing infrastructure
- Secrets are stored in encrypted secret stores or dedicated secrets management services
- Development and production secrets are properly separated
- No sensitive credentials appear in version control history or logs
