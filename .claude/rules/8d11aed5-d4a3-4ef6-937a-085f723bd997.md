<rule_activation id="8d11aed5-d4a3-4ef6-937a-085f723bd997" title="Adopt Secure Secrets Management in CI/CD Pipeline" applies_to="**/*">
These rules are ALWAYS ACTIVE for all files to ensure secure handling of sensitive credentials and API keys within the CI/CD pipeline. Applies to configuration management, status reporting, testing infrastructure, and all components requiring access to secrets.
</rule_activation>

### Rules

- **R-SEC-001** MUST: Not hardcode credentials, API keys, tokens, passwords, or other sensitive data in source code or configuration files committed to version control.
- **R-SEC-002** MUST: Implement a dedicated secrets management system integrated into the CI/CD pipeline that separates secret storage from code (e.g., HashiCorp Vault, AWS Secrets Manager, Azure Key Vault, or CI platform-native solutions).
- **R-SEC-003** MUST: Inject credentials at runtime using environment variables, encrypted secret stores, or secrets management services rather than embedding them in source files.
- **R-SEC-004** MUST: Apply secrets handling patterns consistently across CLI commands, testing infrastructure (including e2e_live.rs), configuration management (config.rs), and status reporting (status.rs).
- **R-SEC-005** SHOULD: Use environment variables as the primary mechanism for runtime credential injection when dedicated secrets management services are not available.
- **R-SEC-006** MUST: Ensure secrets are not exposed through logs, error messages, debug output, or public repositories.
- **R-SEC-007** SHOULD: Enable developers to work with local configurations without requiring access to production secrets through the use of separate development/staging secret stores.
- **R-SEC-008** MUST: Document the secrets management approach and required environment variables for local development setup.

### Verify

```bash
# Check for hardcoded secrets patterns in source code
grep -r -E '(password|api[_-]?key|secret|token|credentials?)\s*=\s*["'\'']((?!["'\'']).{8,})' --include='*.rs' --include='*.toml' --include='*.yaml' --include='*.yml' --include='*.json' . || echo "No hardcoded secrets found"

# Verify environment variable usage for secrets
grep -r 'env::var\|std::env\|get_env' --include='*.rs' config.rs status.rs e2e_live.rs 2>/dev/null && echo "Environment variable usage detected" || echo "WARNING: No environment variable usage found in key files"

# Check for .env files are gitignored
if [ -f .gitignore ]; then
  grep -E '^\.env$|^\.env\..*' .gitignore > /dev/null && echo ".env files properly ignored" || echo "WARNING: .env files may not be gitignored"
fi

# Verify no secrets in git history (sample check on recent commits)
git log -p -S 'password\|api_key\|secret\|token' --all -- '*.rs' '*.toml' '*.yaml' '*.yml' | head -20 | grep -E '(password|api_key|secret|token)\s*=' && echo "WARNING: Potential secrets found in git history" || echo "No obvious secrets in recent git history"

# Check for secrets management configuration
if [ -f "config.rs" ]; then
  grep -E '(Vault|SecretsManager|KeyVault|env::var)' config.rs > /dev/null && echo "Secrets management patterns found in config.rs" || echo "WARNING: No secrets management patterns detected in config.rs"
fi
```

**Accept when:**
- No hardcoded credentials, API keys, tokens, or passwords are present in source code or configuration files
- Secrets are loaded from environment variables or dedicated secrets management services at runtime
- Configuration files (*.toml, *.yaml, *.json) do not contain sensitive credential values
- .env and other local secret files are properly excluded from version control via .gitignore
- Documentation exists describing required environment variables and secrets setup process
- Consistent secrets handling is implemented across config.rs, status.rs, e2e_live.rs, and CLI components
- Error messages and logs do not expose secret values
- Developers can configure local development environments without accessing production secrets
