<rule_activation id="fd1181e0-c600-46af-94d8-829c8e8b2b12" title="Adopt Comprehensive Unit Testing Strategy for CI/CD Pipeline Components" applies_to="src/**">
These rules are ALWAYS ACTIVE for all files matching `src/**` that contain CI/CD pipeline components and build tooling modules.
</rule_activation>

### Rules

- **R-UT-001** MUST: Implement comprehensive unit testing for all CI/CD pipeline components and build tooling modules. Each module in the build, delivery, and tooling infrastructure must include unit tests that validate core functionality in isolation.

- **R-UT-002** MUST: Include unit tests for static analysis components including manifests, languages, and registry functionality.

- **R-UT-003** MUST: Include unit tests for CLI command handlers including sync, auth, and status commands.

- **R-UT-004** MUST: Include unit tests for UI interaction components including confirm dialogs and file confirmation interfaces.

- **R-UT-005** MUST: Include unit tests for generation and tailoring logic.

- **R-UT-006** MUST: Include unit tests for signal processing modules including language resolver, tree-sitter integration, and IR processing.

- **R-UT-007** MUST: Include unit tests for API utilities including retry mechanisms.

- **R-UT-008** MUST: Include unit tests for configuration types and configuration management.

- **R-UT-009** MUST: Co-locate unit tests with source code using `#[cfg(test)]` modules or place them in parallel test directories following Rust's standard testing conventions.

- **R-UT-010** SHOULD: Design tests to validate component contracts and expected behavior, providing clear documentation of functionality.

- **R-UT-011** SHOULD: Focus tests on behavior validation rather than implementation details to reduce brittleness.

- **R-UT-012** SHOULD: Cover edge cases and error conditions in addition to happy path scenarios.

### Verify

**Note:** The source ADR does not contain explicit verification commands. The following verification approach is derived from the testing requirements:

```bash
# Verify unit tests exist for CI/CD components
find src -type f -name '*.rs' | xargs grep -l "#\[cfg(test)\]" || echo "No test modules found"

# Run all unit tests
cargo test --lib

# Check test coverage for critical modules
cargo test --lib -- --nocapture

# Verify test organization follows Rust conventions
find src -type f -name '*.rs' -exec grep -l "mod tests" {} \;
```

**Accept when:**
- All CI/CD pipeline components and build tooling modules have corresponding unit tests
- Static analysis components (manifests, languages, registry) have unit test coverage
- CLI command handlers (sync, auth, status) have unit test coverage
- UI interaction components (confirm dialogs, file confirmations) have unit test coverage
- Generation and tailoring logic has unit test coverage
- Signal processing modules (language resolver, tree-sitter, IR) have unit test coverage
- API utilities (retry mechanisms) have unit test coverage
- Configuration types and management have unit test coverage
- Unit tests are co-located with source code using `#[cfg(test)]` modules or in parallel test directories
- Tests follow Rust's standard testing conventions
- Unit tests validate core functionality in isolation
- Tests provide clear documentation of expected behavior through test cases

<enforcement>
Claude Code MUST NOT skip or defer verification of unit test coverage for CI/CD pipeline components. When modifying or creating CI/CD infrastructure code, unit tests must be implemented before considering the work complete.
</enforcement>
