<rule_activation id="fd1181e0-c600-46af-94d8-829c8e8b2b12" title="Adopt Comprehensive Unit Testing Strategy for CI/CD Pipeline Components" applies_to="src/**">
These rules are ALWAYS ACTIVE for all files in the CI/CD pipeline components and build tooling infrastructure matching `src/**`.

The codebase contains critical CI/CD infrastructure components including static analyzers, CLI commands, API retry logic, configuration management, and signal processing modules. These components form the backbone of the build and delivery pipeline, requiring high reliability and correctness.
</rule_activation>

### Rules

- **R-CICD-001** MUST: Implement comprehensive unit tests for all CI/CD pipeline components and build tooling modules. Each module in the build, delivery, and tooling infrastructure must include unit tests that validate core functionality in isolation.

- **R-CICD-002** MUST: Include unit tests for static analysis components including manifest parsing, language detection, and registry modules.

- **R-CICD-003** MUST: Include unit tests for CLI command handlers including sync, auth, and status commands.

- **R-CICD-004** MUST: Include unit tests for UI interaction components including confirm dialogs and file confirmation interfaces.

- **R-CICD-005** MUST: Include unit tests for generation and tailoring logic modules.

- **R-CICD-006** MUST: Include unit tests for signal processing modules including language resolver, tree-sitter integration, and intermediate representation (IR) handling.

- **R-CICD-007** MUST: Include unit tests for API utilities including retry mechanisms and error handling.

- **R-CICD-008** MUST: Include unit tests for configuration types and management logic.

- **R-CICD-009** SHOULD: Co-locate unit tests with source code using `#[cfg(test)]` modules or place them in parallel test directories following Rust's standard testing conventions.

- **R-CICD-010** MUST: Follow Rust's standard testing conventions with `#[cfg(test)]` modules or separate test files.

### Verify

**Note:** The source ADR does not contain an explicit verify/implementation loop section. The following verification commands are derived from the policy requirements:

```bash
# Check for inline test modules in source files
grep -r "#\[cfg(test)\]" src/ 2>/dev/null | wc -l

# Check for separate test files
find src/ -type f \( -name "*_test.rs" -o -name "tests.rs" \) 2>/dev/null | wc -l

# Check for tests directory
find tests/ -type f -name "*.rs" 2>/dev/null | wc -l

# Run unit tests to verify they pass
cargo test --lib --no-fail-fast 2>&1 | tail -20
```

**Accept when:**

- Each CI/CD pipeline component module (static analyzers, CLI commands, UI components, signal processors, API utilities, configuration types) has corresponding unit tests
- Tests validate core functionality in isolation from external dependencies
- Tests are co-located with source code using `#[cfg(test)]` modules or in parallel test directories
- All unit tests follow Rust's standard testing conventions
- Running `cargo test --lib` executes the test suite and all tests pass
- Tests provide clear documentation of expected behavior through descriptive test names and assertions
- Critical edge cases and error conditions are covered by test cases

<enforcement>
Claude Code MUST NOT skip or defer verification. When modifying CI/CD pipeline components, build tooling, or infrastructure modules, corresponding unit tests MUST be created or updated. All unit tests MUST pass before changes are considered complete.
</enforcement>
